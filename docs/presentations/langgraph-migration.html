Current State — What We Built

A fully custom AI stack — every component written and maintained by the team

ChatManager
Orchestrates the conversation loop — calls the LLM, manages tool call rounds, handles errors
Us
BedrockClient
LLM transport layer — formats messages, sends to AWS Bedrock, parses responses per model
Us
ToolManager
Generates tool definitions from the FastAPI OpenAPI spec; executes HTTP tool calls with auth
Us
MessagePreprocessor
4-stage pipeline that prevents conversation history from overflowing the model's context window
Us
SessionManager
Tracks active chat sessions and conversation history — in-memory, lost on process restart
Us
This architecture was intentionally designed to be modular and self-contained. It has served its purpose well. The natural next step is to replace these custom internals with a mature, maintained framework.

Why LangGraph
6 / 15

Why LangGraph

Five concrete criteria evaluated — LangGraph satisfies all five simultaneously

Criterion 1
Zero new infrastructure
Same AWS Bedrock endpoint, same Claude model, same Postgres instance. LangGraph slots in as an async Python library — no new service, no new database, no new credentials.
Criterion 2
Direct component migration path
Each of the 5 custom components maps one-to-one to a LangGraph primitive. No forced redesign — each is replaced in isolation per phase.
Criterion 3
Capabilities as core primitives
Durable execution, HITL, and parallel subgraphs are first-class graph concepts — not workarounds. Every alternative requires custom scaffolding to approximate them.
Criterion 4
Observability & token tracking included
LangSmith provides full step traces, token usage, and latency per node with zero custom instrumentation — directly addresses two standing requirements.
Criterion 5
Production maturity & commercial backing
Maintained by LangChain Inc. Largest production deployment base of any Python agentic framework as of 2026. Open-source with commercial support. Not a research prototype.
Verdict
No other option qualifies
Amazon Bedrock Agents: no HITL, no durable execution. AutoGen/CrewAI: no Postgres checkpointer. Semantic Kernel: Azure-centric, weaker Bedrock integration. Full rewrite: maximum risk, zero continuity.

Surgical Replacement — What Changes

Five components replaced. One config rename. Everything else untouched.

Current Component Replaced With Why
ChatManager (conversation loop) LangGraph StateGraph Branching, parallel steps, checkpointing, HITL
BedrockClient (LLM transport) ChatBedrockConverse via langchain-aws Same Claude model, no new AWS costs, vendor-maintained
ToolManager (tool execution) ToolNode + explicit tool definitions Testable, composable, full control over tool behavior
MessagePreprocessor (context budget) Custom preprocessing node Preserves existing 4-stage logic, moved into the graph
SessionManager (session lifecycle) langgraph-checkpoint-postgres Durable, persistent, uses existing Postgres instance
BEDROCK_* env vars (81 total) AUTOCHAT_* prefix Removes false AWS coupling; clean rename, no legacy aliases

Multi-job workloads analyzed concurrently via LangGraph subgraphs, eliminating the sequential bottleneck where each job waits for the prior one to complete.
🔭
