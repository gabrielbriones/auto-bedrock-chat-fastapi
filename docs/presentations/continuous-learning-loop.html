02

The Problem

Engineers report three categories of AI reliability issues

🔢
Wrong Math
LLMs "guess" arithmetic instead of computing it. They are text predictors, not calculators.
"IPC = cycles ÷ instructions"
❌ It's instructions ÷ cycles
👻
Fabricated Data
The AI invents metrics, benchmarks, and specifications that don't exist in the source data.
"Job X shows 4.2 GHz frequency"
❌ This job doesn't exist
🔀
Inconsistent Reasoning
Same data produces different analyses. No persistent methodology across sessions.
Normalizes per-core in one response,
uses raw totals in the next
03

Root Causes & What Won't Work

🚫
Won't Fix It
🔄 Switching models — all LLMs share the same failure categories
🎯 Fine-tuning — no training data, no budget, needs periodic re-training
📄 More docs alone — AI retrieves the formula but still does the math wrong
💡
Our Approach
🔧 Deterministic tools for math — code computes, AI interprets
📚 Knowledge Base — domain docs + validated corrections reduce hallucination
🔄 Feedback loop — system learns from expert-reviewed corrections over time

Why Expert Review Is Critical

The gate that prevents feedback from making the system worse

❌
Without Expert Review
👤 User submits wrong correction
▼
⬇️ System auto-adds to knowledge base

✅
With Expert Review
👤 User submits correction (right or wrong)
▼
📋 Enters review queue
▼
🔍 Expert validates, tags, and comments
▼
✅ Only approved corrections reach the KB

Alternatives Evaluated

Technologies and approaches we considered and why they don't fit

🏗️
Amazon Bedrock Knowledge Bases
Managed RAG service. pgvector code is already built — migration would be major effort with no capability gain for feedback workflow.
Future option
🧠
AgentCore Memory
Auto-learns from conversations — including when the AI is wrong. No human review gate. Would amplify errors.
Not a fit
🛡️
Bedrock Guardrails
Detects hallucinations in real-time but doesn't fix them. Complementary, not a replacement. +200-500ms latency.
Future complement
⚔️
LLM Challenger (Multi-AI)
Second model cross-validates responses. Doubles LLM cost and latency. Valuable but deferrable.
Future Phase 4
09

Industry Context

How our approach relates to state-of-the-art AI improvement patterns

🔄
RLHF
OpenAI · Anthropic · Google
Human raters score model outputs. Scores are used to train a reward model, which then fine-tunes the LLM itself via PPO. The model's weights change.
Human feedback Model improves Needs fine-tuning GPU cluster required Weeks per cycle
⛔ Requires massive compute and labeled training data we don't have.
📄 Ouyang et al., "Training language models to follow instructions with human feedback", NeurIPS 2022
📄 Christiano et al., "Deep RL from human preferences", NeurIPS 2017
📄 Bai et al., "Training a Helpful and Harmless Assistant with RLHF", Anthropic 2022
📎
Copilot Studio
Microsoft
Curated knowledge base fed into RAG retrieval. Domain experts manually upload and maintain documents. The LLM itself doesn't change — only the context it receives does.
RAG retrieval Curated KB No GPU needed Manual KB updates No feedback loop
⚠️ KB quality depends entirely on human authors. No learning from usage.
📄 Microsoft Learn, "Knowledge sources in Copilot Studio"
📄 Microsoft Learn, "Use a custom data source for generative answers"
📄 Microsoft, "Build copilots with agent capabilities"
🔗
LangSmith
LangChain · Arize · Weights&Biases
Observability and evaluation platforms. Log every prompt, response, and rating. Surface patterns and regressions. Engineers use data to decide what to fix — manually.
Feedback collection Analytics Regression detection No auto-synthesis Manual action required
⚠️ Great visibility, but fixes still require a human to act on the data.
📄 LangChain, "LangSmith Documentation — Observability & Evaluation"
📄 LangChain, "Evaluate LLM applications with LangSmith"
📄 Arize AI, "LLM Observability & Evaluation"
🎯
Our Approach
Continuous Learning Loop
Combines RAG + observability + automated synthesis. Feedback from trusted users is reviewed by experts, then an LLM turns approved corrections into KB articles that immediately improve future answers.
User feedback Expert review gate Auto KB synthesis Effectiveness tracking No retraining needed
✅ Continuous, safe improvement without model retraining or manual KB authoring.

Inspired by RLHF: We borrow the collect → evaluate → improve loop, but instead of retraining model weights we update the knowledge context — no compute, no data pipeline.
Our differentiator: Expert-reviewed, auto-synthesized KB articles with effectiveness tracking and automatic expiry. No equivalent open-source pattern exists at this level of safety control.
